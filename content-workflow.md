# Web3 Content Workflow

A repeatable process for X posts, articles, and editorial content across projects (dTelecom/dMeetApp, Concrete XYZ, Seismic, and others).

---

## Step 1: Brief

Before writing anything, nail down:

- **Project:** which ecosystem (dTelecom/dMeetApp, Concrete XYZ, Seismic, etc.)
- **Content type:** X post / X thread / editorial article / onboarding copy
- **Goal:** awareness, referral growth, milestone announcement, competitive positioning, community onboarding
- **Audience:** crypto-native degens, general Web3 curious, enterprise-adjacent readers
- **One-line angle:** the single idea the piece hangs on

If any of these are fuzzy, settle them first — everything downstream depends on this.

---

## Step 2: Research

- Pull facts only from **official docs, announcements, or verified sources** (e.g. project docs, official site, official Discord/X announcements) — never guess numbers or mechanics.
- Note anything with hard figures: reward rates, revenue splits, token mechanics, security architecture, tiers.
- Flag anything unverified so it doesn't sneak into the draft as fact.

---

## Step 3: Angle & Hook

- Draft 2–3 title/hook options if it's an article or thread opener.
- Optimize for click-through: lead with tension, contrast, or a specific number — not a vague claim.
- For threads/X posts: the first line has to work standalone (it's what shows in feed).

---

## Step 4: Draft

Match tone to format:

| Format | Tone |
|---|---|
| X post (single) | Punchy, crypto-native, minimal fluff |
| X thread | Narrative arc — hook, build, payoff/CTA |
| Editorial article | Human-sounding, structured, avoids buzzword stacking |
| Onboarding/guide copy | Clear, step-by-step, low jargon |

Write the full draft in one pass — don't over-edit mid-draft.

---

## Step 5: Tone & Humanization Check

Read it back and check:

- [ ] Does this sound like a person, not a press release?
- [ ] Any AI-flavored crutch phrases ("in today's fast-paced world," "unlock the power of")?
- [ ] Is it concise — could any sentence be cut without losing meaning?
- [ ] Does it match the project's established voice (e.g. Concrete's dark-brutalist/technical tone vs dTelecom's crypto-native energy)?

---

## Step 6: Fact-Check Pass

- Cross-check every stat/number/mechanic against the source from Step 2.
- No invented figures, no rounding that changes meaning.

---

## Step 7: Format Pass

- **X post:** under character limit, hook in the first line, hashtags/mentions only if they add reach.
- **Thread:** each tweet stands alone but flows into the next.
- **Article:** headers, scannable structure, CTA at the end.

---

## Step 8: Final Review

- [ ] Brief goal met?
- [ ] Facts verified?
- [ ] Tone matches project?
- [ ] Would *you* stop scrolling for this?

## Step 9: Post/Publish

Ship it. Optionally log what worked (which hook, which angle) so future briefs get faster.

---

### Notes
- Keep a running doc per project of "voice notes" (approved phrases, tone examples) so Step 5 gets quicker over time.
- For milestone/announcement content, always double check the *exact* milestone language from the official source before framing it.
